# Agent UI Library Research

Research date: 2026-07-12. Sources are limited to official documentation, official repositories, and official package manifests.

## Recommendation

Use **assistant-ui** as the default UI foundation, but integrate it through its runtime boundary (`ExternalStoreRuntime`, a custom transport, or a small repository-owned adapter) instead of adopting `@assistant-ui/react-langgraph` immediately. assistant-ui already provides accessible, composable thread/message/composer primitives, Markdown rendering, tool UI, branching, and a real thread-list surface; its architecture explicitly separates rendering, runtime state, backend protocol, and persistence, and supports an application-owned API and database ([architecture](https://www.assistant-ui.com/docs/architecture), [thread component](https://www.assistant-ui.com/docs/ui/thread), [thread-list primitive](https://www.assistant-ui.com/docs/primitives/thread-list), [custom persistence](https://www.assistant-ui.com/docs/integrations/persistence/custom-adapter)). It also explicitly supports React 19 ([React compatibility](https://www.assistant-ui.com/docs/migrations/react-compatibility)).

Do **not** treat the dedicated assistant-ui LangGraph adapter as a drop-in for the repository's current in-process TypeScript LangGraph package. That adapter requires a LangGraph Cloud-compatible API server and uses `@langchain/langgraph-sdk`; it is a good option only if the backend deliberately exposes that API shape ([LangGraph runtime requirements](https://www.assistant-ui.com/docs/runtimes/langgraph/overview), [manual integration and proxy](https://www.assistant-ui.com/docs/runtimes/langgraph/quickstart)).

Choose **CopilotKit** instead if adopting AG-UI as the frontend/backend contract is an intentional architectural decision and first-class generative UI, frontend tools, shared agent state, and human-in-the-loop flows matter more than minimizing runtime coupling. Its runtime officially mounts in Hono and its React frontend communicates with LangGraph or another agent over AG-UI ([architecture](https://docs.copilotkit.ai/concepts/architecture), [AG-UI flow](https://docs.copilotkit.ai/langgraph-fastapi/ag-ui), [generative UI](https://docs.copilotkit.ai/generative-ui)). Durable thread history and cross-device synchronization are not part of the OSS-only persistence story; CopilotKit documents those features as Enterprise Intelligence Platform capabilities ([OSS versus Enterprise](https://docs.copilotkit.ai/concepts/oss-vs-enterprise)).

## Suggested Adoption Sequence

1. Add shadcn/ui/Tailwind only if the team accepts source-owned UI components as the design-system model; both assistant-ui's prebuilt layer and AI Elements use that approach ([assistant-ui architecture](https://www.assistant-ui.com/docs/architecture), [AI Elements README](https://github.com/vercel/ai-elements)).
2. Prototype one assistant-ui thread against a repository-owned mock/external-store runtime, including streamed text, one tool-call card, cancellation, and an error state; this validates the UI without changing the Hono/LangGraph boundary ([runtime architecture](https://www.assistant-ui.com/docs/architecture), [Tool UI](https://www.assistant-ui.com/docs/tools/tool-ui)).
3. Implement conversation-list and history adapters against the repository's own Hono routes and Drizzle schema rather than adopting hosted persistence by default ([assistant-ui custom persistence](https://www.assistant-ui.com/docs/integrations/persistence/custom-adapter)).
4. Run a separate CopilotKit spike only if AG-UI features are required; mount its runtime on Hono and verify the TypeScript LangGraph integration before committing to the protocol or enterprise persistence layer ([CopilotKit architecture](https://docs.copilotkit.ai/concepts/architecture), [OSS versus Enterprise](https://docs.copilotkit.ai/concepts/oss-vs-enterprise)).
